# Resource Allocation Agent

Прототип агента для распределения ресурсов команды геймдев-студии по задачам:
кого, на какой проект, на сколько процентов загрузки поставить — с учётом грейдов,
маржинальности проектов, дедлайнов и истории выполнения задач. Каждый день
парсит данные таск-трекера и тайм-трекинга, выдаёт отчёт в Discord.

Реализовано всё, что было в ТЗ: bench risk, overload, рекомендации по
нераспределённым задачам, маржа план-vs-факт, поиск senior-делает-junior-задачи,
дорогих разработчиков на низкомаржинальных проектах, список свободных людей.

## Ключевое архитектурное решение

**Цифры считает детерминированный код, а не LLM.**

Считать часы, маржу и утилизацию должно ядро на Python. LLM-у даны tools
(`get_bench_risk`, `find_assignees`, `get_all_project_margins` и т.д.), и его
работа — выбрать, какие tools вызвать, и собрать из их вывода человеко-читаемый
отчёт. LLM не складывает часы и не умножает ставки — это арифметика, и доверять
её модели не нужно.

Что мы получаем:
- цифры в отчёте всегда сходятся (нет галлюцинаций в маржах);
- легко тестировать аналитику отдельно от агента;
- если API недоступен — `report.py` строит точно такой же отчёт без LLM
  (см. `build_deterministic_report`);
- LLM можно оставить только там, где он реально нужен: формулировки, сводки,
  свободно-формальные приоритеты ("сегодня важнее закрыть p1 — дедлайн с
  клиентом").

## Алгоритмы

### 1. Загрузка сотрудника (`employee_workload_window`)

Для окна `[today, today + N дней]`:

```
capacity   = working_days_in_window × (weekly_capacity / 5)
committed  = Σ remaining_hours по in-progress / open задачам сотрудника, где
             каждая задача либо целиком попадает в окно (deadline ≤ end),
             либо пропорционально размазывается:
             remaining * (working_days_in_window / working_days_до_дедлайна)
utilization = committed / capacity
```

Пропорциональная разбивка важна: задача на 40 часов с дедлайном через 28 дней
честно отнимает ~20 часов у двухнедельного окна, а не 0 ("дедлайн вне окна")
и не 40 ("давайте всё посчитаем").

### 2. Bench risk

Утилизация в окне 14 дней ниже порога (по умолчанию 0.6).
Группируется по ролям ("3 unity, 2 unreal, 1 frontend") в первой строчке
для быстрого сканирования.

### 3. Overload

Утилизация > 1.0. Дополнительно перечисляются конкретные задачи под риском
дедлайна — это то, что лиду нужно для разговора с PM-ом.

### 4. Маржа проекта (план vs факт)

По таймлогам за последние 30 дней:
```
revenue_actual = Σ (client_rate[grade] × hours) по логам проекта
cost_actual    = Σ (employee.cost_per_hour × hours)
margin_actual  = (revenue - cost) / revenue
delta          = margin_actual - project.target_margin
```

### 5. Senior на junior-задачах (`grade_mismatches`)

Если `GRADE_LEVEL[assignee.grade] - GRADE_LEVEL[task.required_grade_min] ≥ 2`
— сигнализируем. Это прямая потеря маржи: лида или сеньора можно было бы
поставить на более дорогие/сложные задачи.

### 6. Дорогие сотрудники на низкомаржинальном проекте (`expensive_on_low_margin`)

Senior+ с залогированными часами на проекте с `target_margin < 0.20`.
Внутренние инициативы — типичный кейс.

### 7. Скоринг "сотрудник × задача"

```
score = 0.40·role_match
      + 0.20·grade_fit         # 1.0 ровно по грейду, штраф за overshoot
      + 0.15·skill_overlap     # пересечение task.tags и employee.skills
      + 0.15·availability      # доля remaining_hours, влезающая в свободное окно
      + 0.05·past_performance  # avg(spent/estimate) на готовых задачах
      + 0.05·margin_fit        # (rate − cost)/rate vs target_margin проекта
```

Жёсткие фильтры до скоринга: `emp.role == task.required_role` и грейд
не ниже минимального. Скоринг возвращает не только число, но и breakdown
по факторам + текстовый rationale — это нужно для прозрачности и для того,
чтобы LLM мог процитировать причину в отчёте.

Веса заданы явно, чтобы их можно было крутить. Долгосрочно — учить на истории
("какие из подсказанных назначений были приняты тимлидом, какие отклонены").

## LLM-агент (Anthropic tool use)

`src/agent.py` — это классический tool-use loop:

1. Системный промпт (`SYSTEM_PROMPT`) описывает план: какие tools вызвать
   и в каком порядке, какие секции должны быть в отчёте.
2. Цикл `client.messages.create` → если `stop_reason == tool_use`,
   выполняем все tool-вызовы из ответа, складываем их результаты в
   `tool_result`-блоки и шлём обратно. Так — пока модель не вернёт
   `end_turn` с финальным markdown-отчётом.
3. Если `ANTHROPIC_API_KEY` не задан — вызывается `build_deterministic_report`
   и получается тот же отчёт без LLM.

Промпт явно запрещает выдумывать числа — всё, что появляется в отчёте, должно
прийти из tools.

## Запуск

### Быстрый старт через API (рекомендуется)

```bash
pip install -r requirements.txt
cp .env.example .env
# открыть .env, вставить туда ANTHROPIC_API_KEY, опционально DISCORD_WEBHOOK_URL
python run.py
```

Файл `.env` автоматически подхватывается через `python-dotenv`. С ключом
программа использует Claude Sonnet 4.5 для оформления отчёта и сама решает,
какие tools вызывать. Без ключа — всё работает на детерминированной аналитике.

### Без API (offline режим)

```bash
pip install pydantic
python run.py
```

Anthropic SDK и `python-dotenv` не нужны — программа сама определит, что
ключа нет, и пойдёт через `build_deterministic_report`.

### Verbose-режим

```bash
python run.py -v
```

Выведет каждый шаг агента: какие tools вызвал, в каком порядке, с какими
параметрами. Полезно для отладки и для понимания, как работает tool-use loop.

### Постинг в Discord

Добавь в `.env` строку `DISCORD_WEBHOOK_URL=https://discord.com/api/webhooks/...`
(вебхук создаётся в настройках Discord-сервера → Integrations → Webhooks).
`run.py` бьёт длинный отчёт на чанки <2000 символов и постит каждый отдельным
сообщением.

## Обработка ошибок API

В реальной жизни Anthropic API иногда отвечает `529 Overloaded` (часы пик).
В `agent.py` встроен retry с exponential backoff: до 8 попыток с задержкой
1, 2, 4, 8, 16, 32, 60, 60 секунд. Если все попытки провалились — программа
**не падает**, а молча переключается на `build_deterministic_report`. То же
самое происходит при сетевых ошибках, отсутствующем SDK, и т.п. — продакшн
никогда не должен оставлять тимлида без отчёта только потому, что упало
внешнее API.

## Production: что меняется

Для боевой установки нужно заменить `src/data.py` на адаптеры реальных систем.
Сам агент трогать не надо — он работает с `WorldState`.

| Что | Откуда брать в проде |
|---|---|
| `Employee` (id, role, grade, skills, capacity) | HRIS / 1C / справочник людей |
| `cost_per_hour` | финансовая выгрузка (зарплата + накладные) |
| `Project` + `client_rate_per_hour` | финансовый sheet (тот, что у тебя в Google Sheets) |
| `Task` (включая `assignee_id`, `estimate_hours`, `deadline`) | AppTask API |
| `TimeLog` | AppTask тайм-трекинг |

### Парсеры
- AppTask: REST/GraphQL обёртка → маппит её сущности в `Task` / `TimeLog`.
  Парсить **ежедневно** (cron) — задачи без исполнителя, оценки часов, дедлайны.
- Если у вас в Google Sheets лежат планы (как в твоих ссылках) — добавляешь
  `gdrive_adapter.py` через `gspread` или прямой Sheets API.

### Расписание
- Базово: `cron @daily 8:00 → python run.py`. Этого достаточно.
- Сложнее: APScheduler / Celery beat + хранилище состояний — нужно если
  хочется триггерить переотчёт после изменений в AppTask (через webhook).

### Кэширование
Парсинг — самая медленная часть. Сохранять `WorldState` в Postgres / SQLite
с timestamp, агент читает из кэша, обновляется ETL отдельным джобом.

### Безопасность данных
- HR/финансовые данные не должны утекать в LLM провайдера без уведомления.
  В Anthropic есть Zero Data Retention — стоит включить, если контракт
  у студии позволяет. Альтернатива: на проде LLM-цикл выключается
  (отчёт всё равно собирается детерминированно), а LLM используется
  только для свободно-формальных секций.

### Метрики качества
Чтобы улучшать агента, надо собирать обратную связь:
- какие рекомендации тимлид принял / отклонил (фиксировать в БД);
- расхождение `estimate_hours` vs `spent_hours` по задачам (накрутка часов);
- статистика "перетрек / недотрек" по сотруднику и роли (это уже считаем
  в `past_performance`, но нужно сохранять и сравнивать недели).

### Что делать дальше
1. Подключить реальный AppTask API → `src/adapters/apptask.py`.
2. Добавить SQLite/Postgres-кэш + Alembic миграции.
3. Сохранять history of assignments + outcomes — это датасет для обучения
   весов матчинга (grid search или logistic regression на принятых vs
   отклонённых рекомендациях).
4. Webhook-листенер на изменения в AppTask → пересобирает отчёт по триггеру.
5. Добавить prompt-канал в Discord: тимлид пишет "у Бори отпуск с 15-го",
   агент учитывает это в следующем отчёте (через хранилище свободных
   аннотаций, которое тоже подаётся в системный промпт).

## Тестирование

В прототипе нет полноценного pytest-сьюта (это interview-демка), но
структура к нему готова: `analytics.py` и `matching.py` — чистые функции
от `WorldState`, тесты пишутся в одну строку.

Минимальный набор того, что я бы покрыл первым:
- `working_days_between` на границах (вс→пн, праздники);
- `employee_workload_window` для сотрудника без задач, с задачей в окне,
  и с задачей за окном (пропорция);
- `find_best_assignees` отсеивает по роли и по грейду;
- `project_margin_actual` равен `target_margin` ровно когда `cost = rate*(1-target)`;
- end-to-end: `generate_world() → run_agent(state)` возвращает строку с
  ожидаемыми секциями.

## Что покрыто

- ⚠️ Bench risk (низкая загрузка в окне 14 дней)
- 🔥 Overload (перегруз + конкретные задачи под риском дедлайна)
- 💰 Маржа план vs факт по проектам (по тайм-логам)
- Senior на junior-задачах (потеря маржи)
- Дорогие сотрудники на низкомаржинальных проектах
- 🎯 Рекомендации по нераспределённым задачам (с rationale)
- ➕ Частично распределённые задачи (когда на одной задаче несколько исполнителей,
  но их суммарных часов не хватает на закрытие)
- 🌴 Отпуска, больничные, отгулы (`TimeOff`) — уменьшают capacity и availability
- 📅 Государственные праздники (`Holiday`) — вычитаются из рабочих дней
- 🆓 Список свободных людей
- Discord-постинг с разбивкой на чанки <2000 символов
- **Прогноз на горизонт >2 недель.** Текущее окно — 14 дней. Расширяется через
  параметр `days` в tools (он уже есть в `get_bench_risk(days=N)`), но для
  длинного горизонта оценки становятся шумнее, поэтому по умолчанию короткое окно.
